# Dispatch Technical Explanation

This document is a brief technical explanation of the Dispatch project so that a technical reader can quickly understand what the system does, how it runs, and how the main parts connect.

## Project Summary

Dispatch is an internal email platform for sending large campaigns safely through AWS SES.
It manages contacts, lists, segments, templates, campaigns, sending domains, sender profiles, analytics, suppressions, warmup, throttling, and circuit breakers.
The main goal is not just to send email, but to send email safely without damaging domain or IP reputation.

## Main Architecture

```text
Operator
   |
   v
Next.js Frontend
   |
   v
FastAPI Backend API
   |------------------> PostgreSQL
   |------------------> Redis
   |------------------> Cloudflare / Route53
   |
   v
Celery Workers
   |
   v
AWS SES
   |
   v
SNS/Webhook Events -> Event Worker -> PostgreSQL
```

The frontend is the dashboard operators use.
The backend owns business rules and APIs.
Workers handle slow or retryable jobs like sending email, importing CSV files, processing webhooks, and checking domain health.
PostgreSQL stores the main application data.
Redis supports queues, background jobs, and rate/throttle state.
AWS SES is the real email delivery provider.

## Local Development And Docker

Docker is used, but local development does not have to be fully Docker-only.

Common local setup:

```text
Frontend -> pnpm dev
Backend  -> uvicorn
Services -> Docker containers
```

Docker is useful locally for Postgres, Redis, LocalStack, MailHog, and workers.
Docker contains the app or service, but environment variables still configure how it runs.

## Production Deployment

Production is planned around Docker containers and managed AWS services:

```text
Backend API    -> ECS Fargate
Webhook app    -> ECS Fargate
Workers        -> ECS Fargate
Database       -> RDS PostgreSQL
Redis          -> ElastiCache
Email sending  -> AWS SES
Secrets        -> AWS Secrets Manager
DNS automation -> Cloudflare API or Route53
```

In production, the app runs in containers, while AWS manages the database, cache, email delivery, secrets, and infrastructure reliability.

## Cloudflare, DNS, And Domains

Cloudflare is not used to send emails.
It is used to manage DNS records for sending domains.

Dispatch can use the Cloudflare API to create records like:

- SPF
- DKIM
- DMARC
- MAIL FROM
- SES verification records

To automate this, you need a Cloudflare account, a domain using Cloudflare DNS, and a Cloudflare API token with DNS edit permission.
You do not need to buy domains from Cloudflare.
You can buy a domain from Namecheap, GoDaddy, or another registrar, then point the domain's nameservers to Cloudflare.

Nameservers tell the internet where a domain's DNS records are managed.

```text
Buy domain at Namecheap
Set nameservers to Cloudflare
Cloudflare now controls DNS records
Dispatch can add email DNS records through Cloudflare API
```

If Cloudflare is not connected, Dispatch can still show the needed DNS records and an operator can add them manually or through another provider like Route53.

## Secrets And API Keys

The project uses a hybrid model:

```text
App-created API keys -> stored hashed in PostgreSQL
Infrastructure secrets -> env vars locally / AWS Secrets Manager in production
Provider credentials -> env vars or AWS Secrets Manager
```

Examples of infrastructure/provider secrets include AWS credentials, Cloudflare token, Google Postmaster credentials, database URL, Redis URL, and session secrets.
App-created API keys are generated by Dispatch for application access and are stored safely as hashes, not plain text.

## Core Domain Concepts

### Campaign

A campaign is one planned email send, like sending a promo email to a chosen audience.
It connects the sender profile, template version, audience list or segment, generated messages, and analytics.
The campaign is the plan; the messages are the individual emails created for each recipient.

```text
Campaign = Sender Profile + Template + Audience + Schedule
```

### Contacts, Lists, Segments, And Suppressions

A contact is one person or email address.
A list is a saved group of contacts.
A segment is a dynamic audience made from rules.
A suppression is a global do-not-send block.

```text
Contact = the person
List = a saved group
Segment = a filtered group
Suppression = never send to this email
```

Even if a contact is in a list or segment, Dispatch must not send to them if they are suppressed or unsubscribed.

### Domain, Sender Profile, And IP Pool

A domain is the verified sending identity and reputation source, like `example.com` or `news.example.com`.
A sender profile is what the recipient sees in the From field, like `Dispatch News <news@example.com>`.
An IP pool is the SES sending lane or infrastructure group used to send mail.

```text
Domain/subdomain = sending identity and domain reputation
Sender profile   = visible sender name and email
IP pool          = sending infrastructure and IP reputation
SES              = owns and sends through the actual IPs
Dispatch         = chooses which domain/profile/pool to use
```

Many domains or subdomains can use the same IP pool, and one domain can be assigned to a specific pool.
At the start, SES shared IPs are enough.
Dedicated SES IP pools matter later when volume and reputation control become more important.

## Email Sending Flow

Simple flow:

```text
1. Operator creates campaign
2. Dispatch validates safety rules
3. Dispatch freezes the audience into a snapshot
4. Dispatch creates one message per recipient
5. Workers pick queued messages
6. Workers render templates and call AWS SES
7. SES sends the emails
8. SES sends delivery/bounce/complaint/open/click events back
9. Dispatch updates message status, analytics, suppression, and reputation data
```

Before sending, Dispatch checks things like:

- Is the domain verified?
- Is the sender profile valid?
- Is the contact suppressed or unsubscribed?
- Is the domain still in warmup?
- Is the throttle limit safe?
- Is a circuit breaker open?

If something is unsafe, sending should be blocked.
This is the deliverability-first rule.

## Why Workers Matter

The API should not send thousands of emails directly during a request.
Instead, the API prepares and launches the campaign, then workers do the slow work.

Workers are used because email sending can be:

- Slow
- Retried
- Throttled
- Paused
- Blocked by circuit breakers
- Updated later by webhook events

## Webhooks And Analytics

After SES sends an email, events come back through SNS/webhooks.

Examples:

- Delivered
- Bounced
- Complained
- Opened
- Clicked
- Rejected
- Failed rendering

Dispatch stores those events, updates message status, updates campaign analytics, and may add emails to suppression lists.
For example, if a complaint event arrives, Dispatch can mark the message as complained, suppress the email, increase complaint-rate metrics, and possibly trip a circuit breaker.

## Core Safety Idea

Dispatch is built to protect deliverability.
Suppressed or unsubscribed contacts must never receive email.
Warmup and throttling prevent sending too much too fast.
Circuit breakers pause sending when bounce or complaint rates become risky.
Webhook events feed analytics and reputation decisions.

Most important idea:

```text
The API prepares and controls sending.
Workers perform the sending.
AWS SES delivers the email.
Webhooks tell Dispatch what happened.
Dispatch updates analytics and safety systems.
```
